deplow is an **opinionated self-hosted project runtime**. Most apps need a **database**, **object storage**, and a **runtime**. deplow runs that stack on a Docker host you control — with **gVisor** for user apps — so you skip hosted Postgres/Redis/S3 tabs and hand-rolled backup cron.

```text
create empty project
  → add web / worker / postgres / redis services
  → bind apps to data (DATABASE_URL / REDIS_URL)
  → deploy (image | Dockerfile | Railpack) under gVisor
  → URL on *.{baseDomain} via Caddy + cloudflared
  → scheduled Postgres backups → platform MinIO
```

## What you get

<CardGrid stagger>
  <Card title="Service-first stack" icon="rocket">
    Dedicated Postgres and Redis containers per service, shared MinIO with
    per-project buckets — add what you need, bind explicitly.
  </Card>
  <Card title="Build paths" icon="set-up">
    Prebuilt image, Dockerfile, or Railpack from source. Compose deploys
    are not supported.
  </Card>
  <Card title="Secure by default" icon="approve-check">
    User apps run under gVisor with hardened container defaults. Secrets
    encrypted at rest. Docker socket never mounted into project containers.
  </Card>
  <Card title="Backups" icon="document">
    On-demand and scheduled `pg_dump` to the platform backup bucket — no
    separate backup product to wire up.
  </Card>
</CardGrid>

## Explicitly out of scope

See [Scope](/docs/concepts/scope/) for the full list. Short version: not Coolify/Dokploy. No MySQL/Mongo menu, no Compose deploy, no custom domains in v1 (platform wildcard only), no multi-server, no typed CLI.

**v1** includes Caddy + cloudflared wildcard URLs and git **webhooks**. Previews and custom domains are **v2**.
